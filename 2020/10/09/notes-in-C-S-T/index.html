本文最后更新于：2020年10月23日 上午

圆方树的一点注意事项

我们知道，在圆方树中，通常我们会用 TarjanTarjanTarjan 求割点进行缩点，如果你的 TarjanTarjanTarjan 是这样的 (代码有删减)：

void Tarjan(int now){
	dfn[now] = low[now] = ++dep;
	S.push(now);
    
    ...

	for(int i = hdE[now]; i; i = e[i].next){
		if(!dfn[e[i].v]){
			Tarjan(e[i].v);
			low[now] = std::min(low[now], low[e[i].v]);
			if(low[e[i].v] == dfn[now]){ // notice this
				缩点，弹栈
			}
		}
		else{
			low[now] = std::min(low[now], dfn[e[i].v]);
		}
	}
}

那么要注意这个 TarjanTarjanTarjan 有一个小小的细节问题 ：

这种东西也就我这种菜鸡会错 /kk

我们看下面这张图:

假设我们的 DFSDFSDFS 序是这样的：

1→2→4→3→51 \rightarrow 2 \rightarrow 4 \rightarrow 3 \rightarrow 51→2→4→3→5

图整挂了所以 DFS 序有点诡异

然后我们跑 TarjanTarjanTarjan 到 555 时，栈里是这样的：

124351 \quad 2 \quad 4 \quad 3 \quad 512435

然后我们应该这样弹栈 (代码有删减)：

if(low[e[i].v] == dfn[now]){
    ...
    ++cntCube;
    ...
    while(S.top() != e[i].v){
		AddS(cntCube, S.top()); AddS(S.top(), cntCube); S.pop();
        ...
	}
    ...
	AddS(e[i].v, cntCube); AddS(cntCube, e[i].v); S.pop();
    ...
	AddS(cntCube, now); AddS(now, cntCube);
}

原因就是在一些特殊的 DFSDFSDFS 序下，栈中在当前的节点和现在要缩点的子节点之间是会有没有缩掉的点的！

下面是错误示范！！！

if(low[e[i].v] == dfn[now]){
	...
    ++cntCube;
    ...
	while(S.top() != now){
		AddS(cntCube, S.top()); AddS(S.top(), cntCube); S.pop();
        ...
    }
	AddS(cntCube, now); AddS(now, cntCube);
	...
}

这样对应上面的例子，就会把 333 号节点错误地弹出去了！！

虽然这个细节不是很容易出错，但是一旦出错了还是十分致命的 调了俩小时

我认为出现这样的问题的原因是：在 TarjanTarjanTarjan 缩点双时，一个点可能属于多个点双，并且在搜索的过程中，有可能这个节点在一个点双中是搜索路径中间的某个节点，这样会导致如果你在搜索过程中，如果在这个点双中没有回溯完成时，另一个点双已经开始缩点了，就产生了这样错误的删除。
